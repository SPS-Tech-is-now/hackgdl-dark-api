// Zero Trust Network Access

OpenZiti / NetFoundry — ¿qué es?

¿Para qué sirve?

  • OpenZiti — framework open-source de ZTNA (Zero Trust Network Access)
  • NetFoundry — SaaS enterprise sobre OpenZiti: controller + routers gestionados en la nube
  • Crea una red overlay cifrada (fabric) sobre internet — sin abrir puertos de entrada en tus servidores
  • Acceso basado en identidad, no en IP ni en firewall

VPN tradicional

  • Abre un puerto en el firewall → toda la red interna queda alcanzable
  • Cualquiera que llegue al puerto puede intentar conectarse

OpenZiti / NetFoundry

  • 0 puertos abiertos — solo conexiones salientes desde tus servicios
  • Sin identidad ZT enrollada → invisible en la red
  • Política de mínimo privilegio: acceso solo al servicio autorizado
Si el servicio no escucha en ningún puerto y no tiene IP pública, el playbook ofensivo falla antes de empezar.
// Zero Trust Network Access

Cómo funciona el fabric

Nadie escucha. Todos conectan hacia afuera. Sin IP pública, sin puerto abierto — el atacante no tiene dónde apuntar.
// Zero Trust Network Access

NetFoundry — bajo el capó del fabric

🔐 Cifrado

  • TLS 1.3 — sin downgrade posible
  • Forward Secrecy (ECDHE) — tráfico capturado hoy no se puede descifrar mañana
  • AES-256-GCM para el payload del fabric
  • Cifrado end-to-end — los routers no ven el contenido

🪪 Identidades mTLS

  • mTLS mutuo — cliente y servidor presentan certificado; no solo el servidor
  • Certificados ECDSA P-384 por identidad
  • Enrollamiento con token JWT de un solo uso
  • Revocación instantánea desde el Controller — sin reiniciar nada

🌐 Edge Routers globales

  • Americas — US-East, US-West, São Paulo
  • Europa — Frankfurt, Londres, Amsterdam
  • Asia-Pacífico — Singapur, Tokio, Sydney
  • El SDK elige el router más cercano — latencia mínima sin configuración
Este no es tu VPN de empresa. Zero-knowledge routing + mTLS mutuo + TLS 1.3 — interceptar el tráfico no basta: sin la identidad enrollada, el fabric no existe para ti.
// Zero Trust — Lado Servidor

ziti-host: el conector saliente

openziti/docker-compose.yml
services:
  ziti-host:
    image: openziti/ziti-edge-tunnel:latest
    command: run-host \
             --identity /config/crapi-host.json
    volumes:
      - .:/config
    networks:
      - crapi_default
    # SIN sección ports: → 0 puertos de entrada

    # En NetFoundry, el servicio apunta a:
    # hostname: crapi-web (nombre Docker)
    # port: 80

¿Qué hace run-host al iniciar?

  • 1. Lee crapi-host.json — identidad enrollada en NF
  • 2. Abre conexión mTLS SALIENTE → broker NetFoundry
  • 3. Declara en el fabric: "soy el host de crapi-dark, reenvíame a crapi-web:80"
  • 4. Espera solicitudes en el fabric — sin abrir ningún socket local

Verificación en vivo

  • docker exec ziti-host ss -tlnp → vacío
  • nmap ziti-host → 0 puertos abiertos
  • El "puerto" del Dark Service vive en el fabric ZT
El servicio no escucha en ninguna interfaz de red. La conexión hacia crapi-web:80 solo ocurre cuando el fabric ZT la solicita — y solo identidades autorizadas pueden hacerlo.
// Zero Trust — Lado Cliente · modo clásico

ZTHA — ZiTi Host Access

  • Tunneler corre fuera de la app, a nivel OS
  • La app no necesita ningún cambio — tráfico interceptado
  • Ideal para apps existentes que no puedes modificar
  • Desktop Edge Client (Windows / macOS / Linux)
  • ziti-client proxy — localhost:9090
  • ziti-edge-tunnel run — modo tunnel OS
// Zero Trust — Lado Cliente · modo diferenciador ★

ZTAA — ZiTi Application Access

  • SDK embebido dentro de la app — la app es la identidad ZT
  • Sin tunneler externo, sin agente en el OS
  • Funciona en contenedores, serverless, cloud functions
  • En la demo: dark-client.jar + myConsumer.json
  • Sin Desktop App, sin proxy, sin tunneler
  • SDKs: Java · Go · Python · Node.js · .NET
La app es la identidad. Si el host está comprometido pero el proceso no tiene el myConsumer.json enrollado, el fabric ZT no existe para él.

// Implementación

Hazlo tú mismo — OpenZiti en 4 pasos

# 1. Levantar controlador (docker o NetFoundry SaaS)
$ docker run -d openziti/quickstart

# 2. Registrar tu API como dark service
$ ziti edge create service "mi-api" --role-attributes api-service

# 3. SDK en tu app — solo cambiar cómo escucha el servidor
# (Node.js / Python / Go / Java)
$ npm install @openziti/ziti-sdk-nodejs

# 4. Crear identidades con certificados para clientes
$ ziti edge create identity device "mi-cliente" \
    --role-attributes api-clients -o client.jwt
